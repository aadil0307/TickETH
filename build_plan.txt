## TickETH Development Plan

### Phase 1: Smart Contracts (Week 1-2)
This is the foundation — everything depends on the contract interface.

**Decisions to make first:**
- **ERC-721** (1 NFT = 1 unique ticket) vs **ERC-1155** (batch minting, multi-tier in one contract). Recommendation: **ERC-721** for simplicity and clear ownership semantics. ERC-1155 adds complexity we don't need yet.
- **Factory pattern** vs per-event deployment. Recommendation: **Factory contract** — one deployment, clones per event. Saves gas, easier to manage.

**Build order:**
1. `TickETHFactory.sol` — deploys minimal proxy clones per event
2. `TickETHTicket.sol` — ERC-721 with:
   - Tier-based minting (name, price, supply cap)
   - Transfer restriction toggle (organizer-controlled)
   - `Pausable` (emergency stop)
   - `Ownable` (organizer as owner)
   - Metadata URI pointing to IPFS
   - Events emitted for: `TicketMinted`, `TicketTransferred`, `CheckedIn`
3. Unit tests (Hardhat + Chai)
4. Deploy scripts for Mumbai testnet
5. Thirdweb SDK integration testing

**Why first:** Backend can't reconcile mints without contract events. Mobile can't verify ownership. Everything blocks on this.

### Phase 2: Database Schema + Supabase Setup (Week 2)
Run in parallel with late Phase 1.

1. Set up Supabase project
2. Create all 6 tables: `users`, `organizer_requests`, `events`, `ticket_tiers`, `tickets`, `checkin_logs`
3. Add indexes (wallet_address, event_id, token_id, status)
4. Configure Row-Level Security policies per role
5. Set up Supabase Realtime for ticket status changes
6. Add audit log table (`audit_logs` — who did what, when)
7. Seed data for development

**Why now:** Backend needs the schema ready before API development begins.

### Phase 3: Backend Core (Week 3-5)
The central nervous system. Biggest chunk of work.

1. NestJS project setup with TypeScript
2. Wallet-based authentication (sign message → verify → JWT)
3. Role-based guards (Visitor, Attendee, Organizer, Admin, Volunteer)
4. Supabase client integration
5. Environment config (networks, contract addresses, API keys)
6. Error handling + logging foundation

**3B: Core API modules (Week 3-4)**

2. **Organizer module** — request submission, status tracking
3. **Admin module** — approve/reject organizers, system governance, RBAC
4. **Events module** — CRUD, contract deployment trigger, status management
5. **Tickets module** — mint recording, ownership queries, transfer tracking
6. **Tiers module** — pricing, supply management

**3C: Blockchain integration (Week 4)**
1. Chain event listener service (ethers.js / Thirdweb SDK)
   - Listen for `Transfer`, `TicketMinted` events
   - Reconcile on-chain state with DB
2. Mint reconciliation worker (BullMQ)
   - Handle missed events, retries, idempotency
3. Contract interaction service (deploy via factory, read state)
4. RPC provider management (fallback providers, rate limit handling)

**3D: Queue system (Week 4-5)**
1. Redis + BullMQ setup
2. Queues: `mint-reconciliation`, `notifications`, `checkin-processing`, `analytics`
3. Worker processes with retry logic
4. Dead letter queue for failed jobs

**3E: Check-in system (Week 5)**
1. QR payload generation (ticket_id + nonce + expiry + HMAC signature)
2. QR verification endpoint
3. Two-step check-in flow:
   - Volunteer scan → backend validates → push notification to attendee
   - Attendee confirms → wallet signature verified → entry granted
4. Nonce management (single-use, time-bound)
5. Replay protection
6. Offline verification fallback (signed ownership snapshots)

**3F: IPFS integration (Week 5)**
1. Pinata SDK setup
2. Metadata JSON generation (name, description, image, attributes)
3. Upload service for ticket metadata
4. Upload service for event banners (Supabase Storage)

## Key Architectural Decisions to Lock In Before Coding

| Decision | Recommendation | Reason |
|---|---|---|
| Token standard | ERC-721 | Clear 1:1 ticket ownership, simpler |
| Contract pattern | Factory + minimal proxy (EIP-1167) | Gas efficient, one deploy |
| Upgradeability | No proxy upgradeability initially | Simpler, less attack surface |
| State management | Zustand (frontend) | Lighter than Redux for this scope |
| Auth | Wallet signature → JWT | No passwords, Web3-native |
| QR scheme | Backend-signed payload with nonce + expiry | Prevents replay, works offline |
| Metadata | Immutable on IPFS | Once minted, metadata doesn't change |
